If you have ever tried to wire an AI agent into a real piece of software, you know
the hard part isn't the code -- it's **knowing what is true**: what the app can
actually do, which surface to trust, what a write really changed, and how to prove
it. connector-forge is the distilled discipline that makes all of that knowable,
repeatable, and safe.

It's a **skill** (in the Claude Agent Skills / Cowork / Codex sense): a `SKILL.md`
entry point that funnels into a library of focused `references/` and reusable
`assets/templates/`. Host-agnostic, with packs for Claude Code, Cowork, Codex,
Gemini CLI, and Antigravity. Drop it next to your agent and it knows how to walk up
to an unfamiliar system and build a trustworthy bridge to it.

> **The superpower:** point your agent at *any* app and have it forge a connector
> that is verified, governed, and honest -- instead of one that hallucinates an
> endpoint and corrupts your data.

## The method, in one funnel

```
Gate 0.0 Domain Comprehension -> Gate 0 -> Source Registry -> Surface Map
   -> Capability Ledger -> Decision Engine -> Branch
      (CLI / MCP / Hybrid / API / file / browser / STOP)
```

- **Evidence over assumption** -- every capability is `verified`, `hypothesis`, or
  `forbidden`. A backend endpoint *existing* is not a capability.
- **Read first, write governed** -- `read -> simulate -> propose -> approve -> apply ->
  audit`, with readback proving what actually happened.
- **Verification replaces confirmation** -- safety comes from catching reality, not
  from asking the operator to confirm intent five hundred times.
- **A guardian, not a servant** -- the agent may warn that an action looks wrong, but
  it never invents numbers, endpoints, or scope, and never bypasses the app's engine.

## Forged on three real surfaces

This isn't theory. The method was hammered out against three production-grade apps
chosen *because their surfaces could not be more different*. What survived all three
is what made it into the skill.

| Bench | Surface | What it stress-tested |
|---|---|---|
| **Archicad 29** | Desktop CAD -- **3D & vector graphics** via a live add-on | Writing native geometry / layers / zones into a running desktop app; "host loaded" never proving "fresh code"; visual placement as a governed write. |
| **Oracle Primavera P6** | Enterprise scheduling on **Oracle / SQL** | Operating-knowledge gates (which progress method re-estimates which field); schema never wider than the app; destructive-verb discipline on a relational store. |
| **ValuAziende** | **FastAPI + React** web app, native governance | The full governed write ladder; a dynamic AI model registry with a cost guard; Windows / UTF-8 and cloud-sync transport hazards; and multi-agent worksite collaboration with three agents on one bridge. |

Three surfaces -- 3D/vector, relational SQL, and a modern web stack -- and **one
method that held across all of them**. The benches appear here only as sanitized
case studies.

## MCP and CLI, finally one control plane

A real connector is rarely "just MCP" or "just a CLI." The evolution captured here
treats **MCP and CLI as a single integrated control plane**: a CLI surface for
deterministic, scriptable, auditable actions, and an MCP server for rich,
conversational tool use -- sharing the same capability ledger, the same write ladder,
the same readback contract, the same release-claim audit. The `hybrid` branch and
the host packs let that one brain travel across hosts without drift.

## Honesty first

connector-forge is **method and spec guidance** for building connectors. It does not
by itself prove any deployed bridge, package parity, or production readiness -- each is
a separate gate the skill itself defines and demands evidence for. Deliberately **not**
here: any proprietary method, formula, or threshold; any user path, credential,
fingerprint, database name, or private data; any internal log. The skill practices on
itself: nothing is called "ready" without the evidence its own rules require.
